# Phase 13 — Alembic Migrations Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Put the SophiaAI database schema (users, conversations, messages) under Alembic version control so future schema changes are migrations, never a drop-and-recreate.

**Architecture:** Install Alembic, scaffold an `alembic/` directory at the project root, wire `alembic/env.py` to the existing `Base.metadata` from `sophia/db/models.py`, and autogenerate one initial migration that creates the three current tables. The migration is generated against a throwaway empty database so it captures the full `create_table` set even though the dev `sophia_memory.db` already has those tables. A pytest verifies `upgrade head` builds the expected schema on a fresh database and `downgrade base` tears it down.

**Tech Stack:** Alembic 1.x, SQLAlchemy 2.0 (`DeclarativeBase`), SQLite, pytest.

---

## Background You Need Before Starting

Read these before touching anything:

- `sophia/db/models.py` — defines `Base` (a `DeclarativeBase`) and three models: `User` (`users`), `Conversation` (`conversations`), `Message` (`messages`). `Base.metadata` is the single source of truth Alembic must target.
- `sophia/db/database.py:33-34` — the database URL. Default `sqlite:///./sophia_memory.db`, overridable with the `SOPHIA_DB_URL` environment variable. Alembic must read the same env var so dev and migrations never disagree.
- `tests/test_db_service.py:21-28` — the house test pattern: a fresh throwaway SQLite engine per test, schema built with `Base.metadata.create_all`. Phase 13's test follows the same isolation principle but drives Alembic instead of `create_all`.

### Three gotchas specific to this phase

1. **Autogenerate diffs against whatever DB it connects to.** The dev `sophia_memory.db` already contains all three tables (created by `create_all` in Phase 9–12). If you autogenerate while pointed at it, Alembic sees "no difference" and writes an empty migration. Fix: generate the initial migration against a fresh empty throwaway DB so it emits the full `create_table` set. Tasks below do exactly this.
2. **SQLite needs batch mode for future ALTERs.** SQLite cannot `ALTER TABLE ... DROP COLUMN` directly. Alembic's "batch mode" works around this by copy-and-swap. Set `render_as_batch=True` now so the very next migration that alters a column does not blow up. Costs nothing today, saves pain later.
3. **The runtime DB is gitignored and regenerable** (`.gitignore:16-18`). It is machine-local state. You may have a few test conversations in it from Phase 12. They are not precious. We will not delete it — we mark it as already-migrated with `alembic stamp head` (Task 6) so live data survives — but if it ever gets in the way, deleting `sophia_memory.db*` and running `alembic upgrade head` is always safe.

## Task 3: Wire env.py to the models and the real database URL

**Files:**
- Modify: `alembic.ini`
- Modify: `alembic/env.py`

The freshly generated `env.py` follows Alembic's standard template. The edits below match that template's default lines. If a line differs slightly by Alembic version, anchor on the surrounding context.

- [ ] **Step 1: Neutralize the hardcoded URL in alembic.ini**

`alembic init` writes `sqlalchemy.url = driver://user:pass@localhost/dbname` into `alembic.ini`. We set the URL in `env.py` from the environment instead, so leave a harmless placeholder here. Change the line to:

```ini
sqlalchemy.url = sqlite:///./sophia_memory.db
```

This is only a fallback; `env.py` overrides it at runtime (Step 3 below). Setting it to the real default keeps the file honest if anyone reads it.

- [ ] **Step 2: Point target_metadata at Base.metadata**

In `alembic/env.py`, find the default block:

```python
# add your model's MetaData object here
# for 'autogenerate' support
# from myapp import mymodel
# target_metadata = mymodel.Base.metadata
target_metadata = None
```

Replace it with:

```python
# SophiaAI: target the single declarative Base so autogenerate sees all models.
from sophia.db.models import Base

target_metadata = Base.metadata
```

- [ ] **Step 3: Read the database URL from the environment**

In `alembic/env.py`, near the top after the existing `config = context.config` line, add an import for `os` (at the top with the other imports) and override the URL. The default top of `env.py` looks like:

```python
import logging
from logging.config import fileConfig

from sqlalchemy import engine_from_config
from sqlalchemy import pool

from alembic import context

# this is the Alembic Config object...
config = context.config
```

Add `import os` to the import block and, immediately after `config = context.config`, insert:

```python
# SophiaAI: keep migrations and the app pointed at the same database.
# Mirrors sophia/db/database.py — default file DB, overridable via SOPHIA_DB_URL.
config.set_main_option(
    "sqlalchemy.url",
    os.environ.get("SOPHIA_DB_URL", "sqlite:///./sophia_memory.db"),
)
```

- [ ] **Step 4: Enable batch mode for SQLite in offline migrations**

In `alembic/env.py`, find `run_migrations_offline()`. The default body contains:

```python

        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )
```

Add `render_as_batch=True,` inside that call:

```python
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
        render_as_batch=True,
    )
```

- [ ] **Step 5: Enable batch mode for SQLite in online migrations**

In `alembic/env.py`, find `run_migrations_online()`. The default contains:

```python
        context.configure(
            connection=connection, target_metadata=target_metadata
        )
```

Replace it with:

```python
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
            render_as_batch=True,
        )
```

The throwaway scratch DB used in Task 4 (`_alembic_tmp.db`) needs no `.gitignore` entry — the existing `*.db` rule (`.gitignore:14`) already covers it, and Task 4 deletes it immediately after use.

## Task 4: Autogenerate the initial migration against an empty database

The dev `sophia_memory.db` already has the three tables, so we generate against a fresh empty DB to capture the full `create_table` set.

**Files:**
- Create: `alembic/versions/<hash>_initial_schema.py` (Alembic picks the hash)

- [ ] **Step 1: Autogenerate pointed at a throwaway empty DB**

Run (PowerShell — the env var override makes Alembic connect to a brand-new empty file):

```powershell
$env:SOPHIA_DB_URL = "sqlite:///./_alembic_tmp.db"
alembic revision --autogenerate -m "initial schema"
Remove-Item _alembic_tmp.db, _alembic_tmp.db-wal, _alembic_tmp.db-shm -ErrorAction SilentlyContinue
Remove-Item Env:\SOPHIA_DB_URL
```

The first line sets the scratch DB. The second autogenerates by diffing `Base.metadata` against that empty DB. The third deletes the scratch DB and its WAL/SHM sidecars. The fourth clears the env var so the rest of your session uses the real DB.

Expected output during generate includes lines like:
```
INFO  [alembic.autogenerate.compare] Detected added table 'users'
INFO  [alembic.autogenerate.compare] Detected added table 'conversations'
INFO  [alembic.autogenerate.compare] Detected added table 'messages'
  Generating ...alembic\versions\<hash>_initial_schema.py ...  done
```

- [ ] **Step 2: Inspect the generated migration**

Open the new file in `alembic/versions/`. Verify:
- `upgrade()` contains three `op.create_table(...)` calls for `users`, `conversations`, `messages`.
- `users` has `id`, `email` (unique), `hashed_password`, `created_at`.
- `conversations` has `id`, `user_id` (ForeignKey to `users.id`), `title`, `created_at`, `updated_at`.
- `messages` has `id`, `conversation_id` (ForeignKey to `conversations.id`), `role`, `content`, `sources_json` (nullable), `created_at`.
- `downgrade()` contains three matching `op.drop_table(...)` calls (typically in reverse order).

If any table is missing, the env var was not set when you generated — delete the migration file, ensure `_alembic_tmp.db` does not exist, and redo Step 1.

## Task 5: Test that upgrade builds the schema and downgrade tears it down

**Files:**
- Create: `tests/test_migrations.py`

This test drives Alembic programmatically against a fresh temporary SQLite file (never the dev DB), then asserts the resulting tables. It follows the project's isolation principle from `tests/test_db_service.py`.

- [ ] **Step 1: Write the failing test**

Create `tests/test_migrations.py`:

```python
"""Verify the Alembic initial migration builds and drops the full schema.

Mental Model
------------
We run the real migration scripts against a throwaway SQLite file, never the
dev database. ``upgrade head`` must create the three live tables; ``downgrade
base`` must remove them again. This proves the migration is reversible and
matches the models.
"""

from __future__ import annotations

from pathlib import Path

import pytest
from alembic import command
from alembic.config import Config
from sqlalchemy import create_engine, inspect

PROJECT_ROOT = Path(__file__).resolve().parents[1]

EXPECTED_TABLES = {"users", "conversations", "messages"}


@pytest.fixture
def alembic_config(tmp_path: Path) -> Config:
    """Alembic config pointed at a throwaway DB inside the test's tmp dir."""
    db_path = tmp_path / "migration_test.db"
    cfg = Config(str(PROJECT_ROOT / "alembic.ini"))
    cfg.set_main_option("script_location", str(PROJECT_ROOT / "alembic"))
    cfg.set_main_option("sqlalchemy.url", f"sqlite:///{db_path}")
    return cfg


def _table_names(cfg: Config) -> set[str]:
    """Reflect the migrated DB and return its non-alembic table names."""
    url = cfg.get_main_option("sqlalchemy.url")
    engine = create_engine(url, future=True)
    tables = set(inspect(engine).get_table_names())
    engine.dispose()
    return tables - {"alembic_version"}


def test_upgrade_head_creates_all_tables(alembic_config: Config) -> None:
    command.upgrade(alembic_config, "head")
    assert _table_names(alembic_config) == EXPECTED_TABLES


def test_downgrade_base_drops_all_tables(alembic_config: Config) -> None:
    command.upgrade(alembic_config, "head")
    command.downgrade(alembic_config, "base")
    assert _table_names(alembic_config) == set()
```

- [ ] **Step 2: Run the test to confirm it exercises the real migration**

Run:

Expected: both tests PASS. (If you ran this before Task 4 existed, it would FAIL with "Can't locate revision" / no migration — that is the failing-first state. With Task 4 done, it must be green.)

If `test_upgrade_head_creates_all_tables` fails with a missing table, the migration from Task 4 is incomplete — return to Task 4 Step 2.


Expected: all prior tests plus the 2 new migration tests pass (Phase 12 baseline was 144; expect 146 now, allowing for any pre-existing skips).

## Task 6: Reconcile the existing dev database with Alembic

The dev `sophia_memory.db` already has the tables but no `alembic_version` row. We mark it as already-migrated so Alembic does not try to recreate existing tables (which would error), while preserving any live conversation data.

**Files:** none (state change only)

- [ ] **Step 1: Check whether a dev DB exists**

Run:

```powershell

```

- [ ] **Step 2a: If it exists — stamp it as already at head**

Run:

```powershell

```

This writes the current revision into a new `alembic_version` table without running the `create_table` operations, so existing tables and data are untouched. Expected output: `INFO  [alembic.runtime.migration] Running stamp_revision  -> <hash>`.

- [ ] **Step 2b: If it does NOT exist — build it from the migration**

Run:

```powershell

```

This creates `sophia_memory.db` with all three tables via the migration. Expected: `INFO  [alembic.runtime.migration] Running upgrade  -> <hash>, initial schema`.

- [ ] **Step 3: Verify Alembic agrees the DB is current**

Run:

```powershell
alembic current
```

Expected: prints the initial migration's hash followed by `(head)`, e.g. `<hash> (head)`.

- [ ] **Step 4: No commit** — `sophia_memory.db` is gitignored (`.gitignore:16`). This step only reconciles local state.
